## üß± 1Ô∏è‚É£ The React Ecosystem Split

React is split into **two main packages**:

| Package         | Purpose                                                                                  | Where it runs                            |
| --------------- | ---------------------------------------------------------------------------------------- | ---------------------------------------- |
| **`react`**     | Defines *how UI works logically* ‚Äî components, hooks, virtual DOM, state, effects, etc.  | Works *everywhere* (web, mobile, native) |
| **`react-dom`** | Defines *how React connects to the browser DOM* ‚Äî actually renders to `<div id="root">`. | Browser or server (HTML output)          |

In short:

> `react` = Brain üß† (logic, data, state)
> `react-dom` = Hands üñêÔ∏è (puts it on screen in browser)

## üß© 2Ô∏è‚É£ Components ‚Äî What They Really Are

### Definition:

A **component** is just a **function or class** that returns UI (usually JSX).

Example:

```jsx
function Button() {
  return <button>Click Me</button>;
}
```

React calls this function, turns `<button>Click Me</button>` into a *virtual DOM object*,
and then `react-dom` takes that virtual DOM and updates the **real DOM** in the browser.

So:

* **React** manages the *component logic*
* **ReactDOM** manages the *rendering process*

---

## ‚öôÔ∏è 3Ô∏è‚É£ Hooks ‚Äî React‚Äôs Built-in Logic Tools

Hooks are **functions** from the `react` package that give components *React features* like state, lifecycle, context, and refs.

Examples:

| Hook                         | Purpose                           |
| ---------------------------- | --------------------------------- |
| `useState()`                 | Local component state             |
| `useEffect()`                | Side effects (fetch data, timers) |
| `useRef()`                   | Persistent DOM or value reference |
| `useContext()`               | Access shared context             |
| `useMemo()`, `useCallback()` | Performance optimization          |

‚û°Ô∏è Hooks **only exist in `react`** (not `react-dom`),
because they are about *component logic*, not DOM manipulation.

---

## üåê 4Ô∏è‚É£ APIs ‚Äî The Building Blocks

When React docs say **API**, it means **functions or objects you can import** from a package.

Example from `react`:

```js
import { useState, useEffect, createContext } from "react";
```

‚Üí those are **React APIs**.

Example from `react-dom/client`:

```js
import { createRoot } from "react-dom/client";
```

‚Üí that‚Äôs a **ReactDOM API** ‚Äî responsible for *connecting React components to the actual page*.

## üèóÔ∏è 5Ô∏è‚É£ Example: Full Flow

```jsx
// 1. React logic
import React, { useState } from "react";

// 2. React DOM connection
import { createRoot } from "react-dom/client";

// Component (React)
function Counter() {
  const [count, setCount] = useState(0); // Hook from React
  return <button onClick={() => setCount(count + 1)}>Count: {count}</button>;
}

// Mount (ReactDOM)
const root = createRoot(document.getElementById("root"));
root.render(<Counter />);
```

### What happens:

1. `React` defines how `Counter` works (state, re-render).
2. `ReactDOM` finds the `<div id="root">` in HTML.
3. `ReactDOM` mounts the virtual DOM built by React onto the browser DOM.

## üßÆ 6Ô∏è‚É£ What about `react-dom/server`?

That‚Äôs for **server-side rendering (SSR)** ‚Äî it turns your React components into HTML strings *on the server* (Node.js) before sending to the browser.

Example:

```js
import { renderToString } from 'react-dom/server';
import App from './App';

const html = renderToString(<App />);
```

It doesn‚Äôt touch the DOM ‚Äî it just **generates HTML markup**.
